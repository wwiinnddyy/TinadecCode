# TinadecCode 双层智能体编排 v1

## 架构基准

TinadecCode 的产品定位是图形化代理工作台，而不是聊天框桌面壳。核心交互对象是会话、任务图、审批、证据、执行体、上下文包和工作区。C# Core 是唯一智能体状态与策略中枢；Elysia 只做薄网关、OpenAPI 与事件代理；Electron/Vue 只做安全 UI 壳层，不持有业务真相。

当前 MVP 继续沿用 Electron + Vue、Elysia Gateway、C# Agent Core 的三层结构。所有会话状态、模型路由、审批、扩展、智能体配置、任务图与运行事件都必须落在 Core。Gateway 不维护第二套 session/task/agent 状态，Renderer 不接触密钥、文件系统或 shell 能力。

## 双层智能体

规划层包含五类内置智能体：

- 会议智能体：与用户对话，把用户意图拆成任务图、成功标准、依赖和审批点。
- 工具管理智能体：为任务节点选择模型路由、工具、MCP/ACP/Skills 和权限边界。
- 实时上下文压缩智能体：维护上下文包、证据映射、摘要与 token 预算。
- 监督智能体：独立检查目标偏离、权限、预算、质量和取消条件。
- 进化智能体：旁路观察重复模式，生成候选 Skill、MCP、Prompt 或执行体，不进入热路径发布。

执行层由可配置的执行体承接任务。MVP 内置计划执行体、搜索执行体、代码定位执行体、测试执行体、综合模型执行体，以及后续可接入的浏览器/文件/Git、多模态模型、生成模型、本地模型和外部 ACP/CLI 执行体。执行体必须声明模型路由、允许工具、能力标签和审批策略。

## v1 运行时

每次用户消息进入 Core 后，Orchestrator 创建一个 Run，并生成会话级 TaskGraph。会议智能体先产生任务节点；工具管理智能体给节点分派执行层智能体和权限 envelope；监督智能体产出监督发现；上下文压缩智能体产出 ContextPack；进化智能体只在旁路生成候选能力或评估说明。

第一版执行体以结构化 stub 和只读分派为主，重点验证“可见、可审计、可配置”的双层编排，而不是自动修改代码闭环。任何写文件、shell、Git 写入、联网、MCP/ACP 进程启动都继续走审批优先策略。

## 关键对象

- Run：一次用户请求触发的编排运行。
- TaskGraph：会话内可回放的任务图。
- TaskNode：可分派的任务节点，包含目标、成功标准、依赖、风险与状态。
- AgentAssignment：任务节点到执行层智能体的分派记录。
- StepResult：执行体产生的结构化步骤结果与证据。
- ContextPack：上下文压缩智能体产生的摘要、证据映射和压缩比例。
- SupervisionFinding：监督智能体产生的目标、权限、预算或质量检查结果。

## MVP 边界

- 本轮实现任务图、运行、分派、监督和上下文包的可见化与审计，不实现完全自治代码修改。
- 进化智能体候选产物默认 disabled，必须经评测说明和人工启用。
- 多模态模型、生成模型和本地模型先作为执行体 provider 类型与模型路由存在，不实现专用推理运行时。
- Windows-first，Linux 与强企业策略中心后置。
